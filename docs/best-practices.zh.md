# 长图排版最佳实践指南

> 适用平台：X/Twitter（单次发布上限 4 张图）、微博等社交媒体
> 适用语言：中文、日文、英文
> 适用场景：图文混排长图（纵向排列，不横向混排）
> 最后更新：2026-03-29

---

## 核心前提：目标阅读场景

社交媒体长图的传播链路为：**原图 → 平台压缩 → 用户设备渲染**

- **X/Twitter**：图片宽度上限约 1200px，超过会被压缩；手机端展示宽度约 375～430pt
- **微博**：原图上限 20MB，手机端展示宽度约 600～750px
- **读者设备**：须同时兼顾手机（主要）与电脑（次要）两类阅读环境

---

## 一、图片宽度

| 目标场景 | 推荐宽度 | 说明 |
|---|---|---|
| 手机优先（推荐） | **1080px** | 对应主流 iPhone/Android 3x 屏，不触发压缩，清晰度充裕 |
| 电脑兼顾 | **1200px** | X/Twitter 无损上限，适合宽屏展示 |
| 极致清晰 | 不建议超过 1200px | X 会压至 1200px，微博亦会压缩，得不偿失 |

**结论：1080px 为黄金宽度**，兼顾手机原生分辨率与平台无损传输。若内容主要面向 PC 端读者，可选 1200px。

---

## 二、正文字号

基准画布宽度：1080px

| 语言 | 推荐字号 | 说明 |
|---|---|---|
| **中文** | **32～36px** | 汉字字形密度高，视觉重量大，需较大字号保证舒适度 |
| **日文** | **30～34px** | 假名与汉字混排，假名较轻盈，可比中文略小 1～2px |
| **英文** | **28～32px** | 拉丁字母以 x-height 为标准，Noto Sans/Serif 在此区间清晰 |

> **换算验证**：1080px 图片在手机（375pt 宽）上缩放比约为 2.88×，即 32px 图片字 ≈ 手机实际 ~11pt，接近正文舒适阅读阈值（≥10pt）。

---

## 三、标题字号体系

采用**模块化比例尺（Modular Scale）**，比率区间 **1.15～1.75**。比率过大会导致标题在长图中显得突兀臃肿。

以正文为基准向上倒推：

| 级别 | 倍率 | 中文（基准 32px） | 日文（基准 30px） | 英文（基准 28px） |
|---|---|---|---|---|
| 正文 Body | 1× | 32px | 30px | 28px |
| H5 五级标题 | 1.15× | **36px** | **34px** | **32px** |
| H4 四级标题 | 1.25× | **40px** | **38px** | **35px** |
| H3 三级标题 | 1.333× | **48px** | **44px** | **40px** |
| H2 二级标题 | 1.5× | **56px** | **52px** | **48px** |
| H1 一级标题 | 1.75× | **72px** | **64px** | **56px** |

> **实操建议**：长图内若无五级标题的实际需求，可将 H4/H5 合并处理，避免层次感模糊。标题层级不宜超过 3 级。

---

## 四、页面边距

画布宽度 1080px 时：

| 位置 | 推荐值 | 说明 |
|---|---|---|
| **左边距** | **80～96px** | 有效内容宽度约 900px，维持约 8.3% 留白率 |
| **右边距** | **80～96px** | 与左侧对称，避免文字顶边 |
| **顶部边距** | **100～120px** | 容纳品牌标识或视觉呼吸区，防止手机截图被系统 UI 裁切 |
| **底部边距** | **80～100px** | 略小于顶部，视觉上更"稳" |

有效排版宽度：1080 - 80×2 = **920px**（推荐值），不超过 960px。

---

## 五、段落间距

| 场景 | 推荐值 | 说明 |
|---|---|---|
| **正文段落间距** | **1.0～1.5× 行高**（约 52～64px） | 保证段落呼吸感，中文尤其重要，帮助读者识别段落边界 |
| **标题之后接正文** | **0.75× 标题行高** | 将标题与其从属正文视觉绑定，体现层级归属 |
| **正文之后接新标题** | **1.5～2× 正文行高**（约 80～96px） | 与上方内容切割，给新标题留足呼吸空间 |

---

## 六、行高

| 级别 | 行高倍率 | 中文实际行高（基准 32px） | 英文实际行高（基准 28px） |
|---|---|---|---|
| 正文 Body | **1.7～1.8×** | 32×1.75 ≈ **56px** | 28×1.75 ≈ **49px** |
| H5 五级标题 | **1.5×** | 36×1.5 = **54px** | 32×1.5 = **48px** |
| H4 四级标题 | **1.5×** | 40×1.5 = **60px** | 35×1.5 ≈ **52px** |
| H3 三级标题 | **1.4×** | 48×1.4 ≈ **67px** | 40×1.4 = **56px** |
| H2 二级标题 | **1.3×** | 56×1.3 ≈ **73px** | 48×1.3 ≈ **62px** |
| H1 一级标题 | **1.2～1.25×** | 72×1.25 = **90px** | 56×1.2 ≈ **67px** |

> **中文行高必须 ≥1.7×**：汉字无 descender（字母下伸部分），密排时易视觉粘连，行高是中文阅读舒适度的首要调控参数。

---

## 七、字间距（Letter Spacing）

| 语言 | 级别 | 推荐字间距 | 说明 |
|---|---|---|---|
| **中文** | 正文 | `0.05em`（约 1.6px @ 32px） | 轻微松动，减少汉字密排压迫感 |
| **中文** | H1～H2 标题 | `0.01～0.02em` | 标题字号大，不宜过松 |
| **日文** | 正文 | `0.05em` | 与中文相似处理 |
| **日文** | 标题 | `0～0.02em` | 依字形疏密决定，假名多则可略松 |
| **英文** | 正文 | `0`（默认） | Noto 字体本身 Kerning 已优化，无需干预 |
| **英文** | 全大写标题 | `0.05～0.1em` | 大写字母字距需人工补偿 |
| **英文** | H1～H2 混合大小写 | `-0.01～0em` | 大字号字距应轻微收紧，避免松散 |

## 八、基于大模型 Agent 的主题配色策略

> **排版铁律**：无论内容类型和风格如何变化，**排版物理参数（字号、行高、间距等）必须保持绝对稳定**。系统绝不可以因内容不同而改变字号。Agent 唯一被授权自适应调整的视觉参数，仅限于**色彩体系（背景色、文本色、边框色）**。

### 1. Agent 内容判别策略

在进入排版渲染引擎前，大模型 Agent 对 Markdown 内容进行极速分类判别策略：

- **输入**：全文或前 800 个字符（足够提取核心语义信息及排版特征）。
- **研判维度**：
  - **词汇特征**：含大量专业术语、代码块、数据或英文缩写 → **科技类**
  - **语气特征**：客观冷静陈述 → **科技类**；长句叙述、知识科普 → **人文类**；抒发情绪、第一人称随需、短句多 → **情感类**
  - **版式特征**：含密集多层列表或复杂表格 → **科技类**
- **输出约束**：Agent 必须仅返回 JSON 枚举值 `{"theme": "tech" | "humanities" | "emotion"}`，不生成任何废话。

### 2. 三种开箱即用的色彩主题

渲染管线收到上述枚举值后，注入对应的 CSS 颜色令牌（Design Tokens）：

#### 主题 A：科技与资讯类（Tech & News）
- **风格调性**：干净、客观、锐意，适合高信息密度阅读。
- **色值规范**：
  - **背景色（冷白）**：`#F8F9FA`
  - **主干文本（标题/正文）**：`#1A1B1E`（极深灰，禁止纯黑）
  - **次级文本（元数据/引用）**：`#5C5F66`
  - **装饰线（边框/分割线）**：`#E9ECEF`

#### 主题 B：人文与叙述类（Humanities & Narrative）
- **风格调性**：经典、沉静、如传统出版物般的纸张质感，长文阅读极度友好。
- **色值规范**：
  - **背景色（暖白）**：`#FAF8F5`
  - **主干文本（标题/正文）**：`#2C2B29`
  - **次级文本（元数据/引用）**：`#73706B`
  - **装饰线（边框/分割线）**：`#E6E2DA`

#### 主题 C：情感与生活类（Emotion & Life）
- **风格调性**：松弛、亲切、带呼吸感与温度感，契合日记与散文随笔。
- **色值规范**：
  - **背景色（奶油黄）**：`#FDF6E3`
  - **主干文本（标题/正文）**：`#332F2A`
  - **次级文本（元数据/引用）**：`#857C70`
  - **装饰线（边框/分割线）**：`#EBE1C5`

> **提示**：深色模式（Dark Mode）由于高度涉及用户的物理阅读环境偏好（白天/夜间），不应由 Agent 推测，应设计为供使用方端侧手动强制切换的独立配置。无论哪种主题，**都强烈建议避免纯白背景 `#FFFFFF`**，不仅对比度刺眼，在手机开屏中也缺乏边界阻断感。

## 十一、X/Twitter 多图切分策略

### 平台约束

| 约束项 | 限制值 |
|---|---|
| 单次发布最多图片数 | **4 张** |
| 单张图片文件大小上限 | **5MB**（PNG） |
| 画布宽度基准 | **1080px** |

---

### 单张图片高度上限（H_max）

| 内容类型 | 推荐 H_max | 说明 |
|---|---|---|
| 纯文字 / 文字为主 | **10000px** | 1080×10000 PNG ≈ 1～2MB，文字压缩比高 |
| 图文混排（含嵌入图片） | **8000px** | 图片像素不可压缩，保留安全边际 |
| **保守默认值（推荐）** | **8000px** | 统一使用此值，无需按内容类型区分 |

**总容量上限 = H_max × 4 = 32000px**

---

### 张数决策公式

以实际渲染高度 `H_total` 为输入：

```
N = ceil(H_total / H_max)

N = 1  →  单图，无需切分           （H_total ≤  8000px）
N = 2  →  切为 2 张               （H_total ≤ 16000px）
N = 3  →  切为 3 张               （H_total ≤ 24000px）
N = 4  →  切为 4 张               （H_total ≤ 32000px）
N > 4  →  ERROR：内容超出容量上限，拒绝渲染
```

---

### 切分点选取算法

**理想目标切点**（理论均分）：

```
target_y[i] = H_total × i / N    （i = 1 … N-1）
```

**实际切点**：在目标切点附近搜索最近的**语义边界**，优先级从高到低：

| 优先级 | 语义边界类型 | 说明 |
|---|---|---|
| 1（最佳） | H1 / H2 标题的**上边缘** | 语义最完整，章节自然断点 |
| 2 | H3 / H4 标题的上边缘 | 次级章节边界 |
| 3 | 段落之间的**间隙中点** | 常规段落边界 |
| 4 | 嵌入图片的**下边缘** | 不在图片内部切断 |
| 5（兜底） | 行与行之间的间隙 | 最后手段，保证不硬切文字 |

**搜索窗口规则**：
- 初始搜索窗口：目标切点 ±15% × (H_total / N)
- 若窗口内无语义边界：扩展至 ±25%
- 若依然无边界（极罕见，如超长单段落）：按行间隙兜底

**切分禁止规则**（铁律）：
- 禁止在**段落内部**切断（同一 `<p>` 标签内不可切）
- 禁止在**嵌入图片内部**切断
- 禁止在**表格内部**切断（表格须整体归属某一张图）
- 禁止在**代码块内部**切断

**超大元素越界回退策略（Oversized Fallback）**：
若系统排布到某个巨大图片（或超长代码块），其本身高度已撑满甚至超出了 `±25%` 的搜索窗口，导致整个窗口内查无语义边界（比如一整张 5000px 的配图刚好跨过 8000px 切断点），必须启动降级策略以破除死锁：
1. **优先向上回退（推荐）**：忽略原定切点，向上无限扩大搜索范围，直到触碰该巨幅图片的**上边缘**。虽然这会导致当前切片总高度变短（例如提早在 5000px 处切图），但能100%保全图片完整性。
2. **向下拓展容忍**：如果向上回退导致上一张图片过短（如切成一张只有 2000px 的碎片图，十分难看），则向下寻找图片的**下边缘**。条件是切出后的图片高度不可突破 10000px 绝对红线。
3. **暴力硬截断（终极兜底）**：若一幅单图的高度本身就已经超过了平台绝对上限（即 `H > 10000px` 的极端条漫），此时「禁止在图片/代码块内部切断」的铁律必须被打破失效，系统在此处直接无视元素结构，在 8000px 处执行物理硬切断，以防出现发布失败灾难。

---

### 两阶段容量预检机制

为避免全量渲染完成后才发现超限（浪费渲染资源），实施两阶段检测：

#### 第一阶段：预估检测（渲染前，毫秒级）

基于 Markdown 源文件做快速统计估算：

| 元素类型 | 估算高度 |
|---|---|
| 每行正文（920px 宽 / 32px 字号，约 28 字/行） | 56px（含行高） |
| 每个段落间距 | 56px |
| 每张 H1 标题（含前后间距） | 90 + 108 + 40 = 238px |
| 每张 H2 标题（含前后间距） | 73 + 96 + 32 = 201px |
| 远程 `http` 网络图片 | **直接拦截报错**（当前骨架版不支持网络图片的稳定性预测） |
| 本地相对路径图片 | 按照「**按宽缩放，绝不放大**」准则：如果原图宽超 920px 则按 `实际高度×(920/实际宽度)` 压算；若原图宽 <= 920px，则取真实物理 `高度` 不做任何拉升。毫秒级读取 Header，真实精确。 |
| 顶部 + 底部边距 | 108 + 88 = 196px |

**预估公式**：

```
H_estimated = 196（固定边距）
            + Σ 各行正文高度
            + Σ 各段落间距
            + Σ 各标题高度
            + Σ 嵌入图片估算高度
```

**预检门槛**：`H_estimated > 32000 × 0.75`（即 24000px）时触发警告；`> 32000px` 时直接报错，跳过渲染。

> 0.75 系数是因为预估偏保守，给实际渲染留有余量。若预估超过 24000px，进入渲染后仍以实际高度为最终裁判。

#### 第二阶段：精确检测（渲染后）

- Playwright 完成截图后，读取实际页面高度 `page.height`
- 若 `page.height > 32000px` → 报错并清理所有临时文件
- 错误信息应包含：实际高度、超出量、建议削减的内容估算字数

---

### 报错信息规范

超出容量上限时，系统应输出结构化错误，示例：

```
ERROR [CAPACITY_EXCEEDED]
  实际渲染高度:  36480px
  容量上限:      32000px（4 张 × 8000px）
  超出量:        4480px（约 14%）
```
