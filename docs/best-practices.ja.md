# 長編画像組版ベストプラクティス

> 対応プラットフォーム：X/Twitter（1投稿最大4枚）、微博、その他ソーシャルメディア
> 対応言語：中国語、日本語、英語
> 対応シナリオ：テキスト・画像混在長編画像（縦方向レイアウトのみ）
> 最終更新：2026-03-29

---

## 核心前提：目標閲覧シナリオ

ソーシャルメディア長編画像の配信チェーン：**原画像 → プラットフォーム圧縮 → ユーザーデバイスレンダリング**

- **X/Twitter**：画像幅の上限は約1200px。スマートフォン表示幅は約375〜430pt
- **微博**：原画像上限20MB。スマートフォン表示幅は約600〜750px
- **読者デバイス**：スマートフォン（主）とPC（副）の両方の閲覧環境に対応が必要

---

## I. 画像幅

| 対象シナリオ | 推奨幅 | 説明 |
|------------|--------|------|
| モバイル優先（推奨） | **1080px** | 主流iPhone/Android 3xスクリーンに対応、圧縮なし、十分な解像度 |
| PC兼用 | **1200px** | X/Twitter無損上限、ワイドスクリーン表示に最適 |
| 超高解像度 | 1200px超は非推奨 | Xは1200pxに圧縮、微博も圧縮するため効果なし |

**結論：1080pxが黄金の幅**。モバイルのネイティブ解像度とプラットフォームの無損転送を両立。PC向けコンテンツが主な場合は1200pxも可。

---

## II. 本文フォントサイズ

ベースキャンバス幅：1080px

| 言語 | 推奨サイズ | 説明 |
|------|-----------|------|
| **中国語** | **32〜36px** | 漢字の字形密度が高く視覚的重量が大きいため、大きめのサイズが必要 |
| **日本語** | **30〜34px** | 仮名と漢字の混在。仮名は軽いため中国語より1〜2px小さくしても良い |
| **英語** | **28〜32px** | ラテン文字はx-highを基準に、Noto Sansはこの範囲で明瞭 |

> **換算確認**：1080px画像のスマートフォン（375pt幅）での縮小比は約2.88×。つまり32pxの画像文字 ≈ スマートフォン実際の〜11pt、快適閲覧閾値（≥10pt）に近い。

---

## III. 見出しフォントサイズシステム

**モジュラースケール**を採用し、比率範囲は **1.15〜1.75**。比率が大きすぎると長編画像で見出しが突出して見える。

本文を基準に上方向に算出：

| レベル | 倍率 | 中国語（基準32px） | 日本語（基準30px） | 英語（基準28px） |
|-------|------|-----------------|-----------------|----------------|
| Body | 1× | 32px | 30px | 28px |
| H5 | 1.15× | **36px** | **34px** | **32px** |
| H4 | 1.25× | **40px** | **38px** | **35px** |
| H3 | 1.333× | **48px** | **44px** | **40px** |
| H2 | 1.5× | **56px** | **52px** | **48px** |
| H1 | 1.75× | **72px** | **64px** | **56px** |

> **実践アドバイス**：5レベル見出しを実際に使用する必要がない場合、H4/H5を統合して階層の曖昧さを避けることを推奨。見出し階層は3レベル以下が望ましい。

## IV. ページ余白

キャンバス幅1080pxの場合：

| 位置 | 推奨値 | 説明 |
|------|--------|------|
| **左余白** | **80〜96px** | 有効コンテンツ幅は約900px、約8.3%の余白率を維持 |
| **右余白** | **80〜96px** | 左側と対称、テキストが端に触れるのを防ぐ |
| **上余白** | **100〜120px** | ブランド識別または視覚的余裕スペースを確保、スマートフォンのスクリーンショットがシステムUIにクロップされるのを防ぐ |
| **下余白** | **80〜100px** | 上余白より若干小さく、視覚的に安定感を与える |

有効レイアウト幅：1080 - 80×2 = **920px**（推奨）、960px以下。

---

## V. 段落スペーシング

| シナリオ | 推奨値 | 説明 |
|---------|--------|------|
| **本文段落間** | **1.0〜1.5×行高**（約52〜64px） | 段落の呼吸感を確保 — 中国語では特に重要 |
| **見出し後の本文前** | **0.75×見出し行高** | 見出しをその従属コンテンツに視覚的に結びつける |
| **本文後の新見出し前** | **1.5〜2×本文行高**（約80〜96px） | 上のコンテンツと区切り、新しい見出しに余裕を与える |

---

## VI. 行間

| レベル | 行高比率 | 中国語実際値（基準32px） | 英語実際値（基準28px） |
|-------|--------|-------------------|----------------|
| Body | **1.7〜1.8×** | 32×1.75 ≈ **56px** | 28×1.75 ≈ **49px** |
| H5 | **1.5×** | 36×1.5 = **54px** | 32×1.5 = **48px** |
| H4 | **1.5×** | 40×1.5 = **60px** | 35×1.5 ≈ **52px** |
| H3 | **1.4×** | 48×1.4 ≈ **67px** | 40×1.4 = **56px** |
| H2 | **1.3×** | 56×1.3 ≈ **73px** | 48×1.3 ≈ **62px** |
| H1 | **1.2〜1.25×** | 72×1.25 = **90px** | 56×1.2 ≈ **67px** |

> **中国語の行高は必ず≥1.7×**：漢字にはディセンダー（文字の下伸部分）がなく、密設定では視覚的な粘連が起きやすい。行高は中国語閲覧の快適さを調整する最重要パラメータ。

---

## VII. 字間隔（Letter Spacing）

| 言語 | レベル | 推奨字間隔 | 説明 |
|-----|-------|----------|------|
| **中国語** | 本文 | `0.05em`（32pxで約1.6px） | 微かな緩和で漢字密設定の圧迫感を軽減 |
| **中国語** | H1〜H2見出し | `0.01〜0.02em` | 見出しは字サイズが大きいため、緩めすぎない |
| **日本語** | 本文 | `0.05em` | 中国語と同様の処理 |
| **日本語** | 見出し | `0〜0.02em` | 字形の密度による。仮名が多い場合は若干緩める |
| **英語** | 本文 | `0`（デフォルト） | Notoフォントのカーニングは最適化済み、調整不要 |
| **英語** | 全大文字見出し | `0.05〜0.1em` | 大文字の字間隔は手動補正が必要 |
| **英語** | H1〜H2 混合大小文字 | `-0.01〜0em` | 大きいフォントサイズは若干引き締める |

## VIII. 大規模言語モデル Agent ベースのテーマカラー戦略

> **組版鉄則**：コンテンツの種類やスタイルがどのように変化しても、**組版の物理パラメータ（フォントサイズ・行高・間隔など）は絶対的に安定していなければならない**。システムはコンテンツによってフォントサイズを変えてはならない。Agentが適応的に調整を授権された唯一の視覚パラメータは**カラーシステム（背景色・テキスト色・ボーダー色）**のみ。

### 1. Agent コンテンツ分類戦略

組版レンダリングエンジンに入る前に、LLM AgentがMarkdownコンテンツを超高速分類：

- **入力**：全文または先頭800文字（コアセマンティック情報と組版特性の抽出に十分）。
- **分類次元**：
  - **語彙特性**：専門用語・コードブロック・データ・英語略語が多い → **テクノロジー**
  - **語調特性**：客観的・冷静な叙述 → **テクノロジー**；長文叙述・知識普及 → **人文**；感情表現・一人称・短文多い → **エモーション**
  - **レイアウト特性**：多層リストや複雑なテーブルが多い → **テクノロジー**
- **出力制約**：AgentはJSONの列挙値 `{"theme": "tech" | "humanities" | "emotion"}` のみを返す。余分な内容は含めない。

### 2. 3種類のすぐに使えるカラーテーマ

レンダリングパイプラインが列挙値を受け取った後、対応するCSSカラートークン（Design Tokens）を注入：

#### テーマA：テクノロジー・ニュース類
- **スタイル**：クリーン・客観・鋭い — 高情報密度の閲覧に適切。
- **色値**：
  - **背景色（クールホワイト）**：`#F8F9FA`
  - **主要テキスト（見出し/本文）**：`#1A1B1E`（極めて濃いグレー — 純黒は禁止）
  - **補助テキスト（メタデータ/引用）**：`#5C5F66`
  - **装飾線（ボーダー/区切り線）**：`#E9ECEF`

#### テーマB：人文・叙述類
- **スタイル**：クラシック・落ち着き・伝統的な出版物のような紙質感 — 長文閲覧に極めて友好的。
- **色値**：
  - **背景色（ウォームホワイト）**：`#FAF8F5`
  - **主要テキスト（見出し/本文）**：`#2C2B29`
  - **補助テキスト（メタデータ/引用）**：`#73706B`
  - **装飾線（ボーダー/区切り線）**：`#E6E2DA`

#### テーマC：エモーション・ライフ類
- **スタイル**：リラックス・親しみやすい・呼吸感と温かみを持つ — 日記やエッセイに適切。
- **色値**：
  - **背景色（クリームイエロー）**：`#FDF6E3`
  - **主要テキスト（見出し/本文）**：`#332F2A`
  - **補助テキスト（メタデータ/引用）**：`#857C70`
  - **装飾線（ボーダー/区切り線）**：`#EBE1C5`

> **注意**：ダークモードはユーザーの物理的な閲覧環境の好み（昼/夜）に大きく関わるため、Agentが推測するべきではなく、クライアント側で手動に強制切換する独立設定として設計すべき。どのテーマでも**純白背景 `#FFFFFF` は強く非推奨**：コントラストが強すぎて目に刺さり、スマートフォン表示時に境界感も欠く。

## XI. X/Twitter マルチ画像分割戦略

### プラットフォーム制約

| 制約項目 | 上限値 |
|---------|-------|
| 1投稿あたりの最大画像数 | **4枚** |
| 1枚の画像ファイルサイズ上限 | **5MB**（PNG） |
| キャンバス幅基準 | **1080px** |

---

### 単一画像高さ上限（H_max）

| コンテンツタイプ | 推奨H_max | 説明 |
|--------------|---------|------|
| テキストのみ / テキスト主体 | **10000px** | 1080×10000 PNG ≈ 1〜2MB、テキスト圧縮率が高い |
| 画像混在（埋め込み画像あり） | **8000px** | 画像ピクセルは圧縮不可、安全余裕を確保 |
| **保守的デフォルト（推奨）** | **8000px** | 統一使用、コンテンツタイプで区別不要 |

**総容量上限 = H_max × 4 = 32000px**

---

### タイル数決定式

実際のレンダリング高さ `H_total` を入力として：

```
N = ceil(H_total / H_max)

N = 1  →  単一画像、分割不要             (H_total ≤  8000px)
N = 2  →  2枚に分割                     (H_total ≤ 16000px)
N = 3  →  3枚に分割                     (H_total ≤ 24000px)
N = 4  →  4枚に分割                     (H_total ≤ 32000px)
N > 4  →  ERROR：コンテンツが容量上限を超過、レンダリング拒否
```

---

### カットポイント選択アルゴリズム

**理想ターゲットカットポイント**（理論的均等分割）：

```
target_y[i] = H_total × i / N    (i = 1 … N-1)
```

**実際のカットポイント**：ターゲット付近で最も近い**セマンティック境界**を検索、優先順位（高→低）：

| 優先度 | セマンティック境界タイプ | 説明 |
|-------|-------------------|------|
| 1（最良） | H1 / H2見出しの**上端** | 最も完全なセマンティクス、自然な章の区切り |
| 2 | H3 / H4見出しの上端 | サブセクションの境界 |
| 3 | 段落間の**ギャップ中点** | 通常の段落境界 |
| 4 | 埋め込み画像の**下端** | 画像内部での切断を避ける |
| 5（フォールバック） | 行と行の間のギャップ | 最終手段 — テキストをハードカットしないことを保証 |

**検索ウィンドウ規則**：
- 初期検索ウィンドウ：ターゲットカットポイント ±15% × (H_total / N)
- セマンティック境界が見つからない場合：±25% に拡張
- それでも境界なし（非常にまれ）：行ギャップにフォールバック

**カット禁止規則**（鉄則）：
- 禁止：**段落内部**でのカット（同一 `<p>` タグ内はカット不可）
- 禁止：**埋め込み画像内部**でのカット
- 禁止：**テーブル内部**でのカット（テーブルは1枚の画像に完全に属すること）
- 禁止：**コードブロック内部**でのカット

**越境デッドロック解除（Oversized Fallback）**：
巨大な画像（または非常に長いコードブロック）が ±25% 検索ウィンドウを満たすかそれを超え、ウィンドウ内にセマンティック境界が存在しない場合（例：5000pxの画像が8000pxのカットポイントを跨ぐ）、以下の降級戦略を起動：
1. **上方退避（推奨）**：元のカットポイントを無視し、巨大画像の**上端**に到達するまで上方向に検索を拡大。現在のタイルが短くなる可能性があるが（例：8000pxではなく5000pxでカット）、画像の完全性を100%保全。
2. **下方延伸容認**：上方退避によりタイルが短すぎる場合（例：2000pxのみ — 視覚的に貧しい）画像の**下端**を下方向に検索。条件：結果のタイル高さが10000px絶対上限を超えないこと。
3. **強制ハードカット（最終手段）**：単一画像の高さ自体がプラットフォームの絶対上限を超える場合（`H > 10000px` — 極端な縦スクロール漫画形式）、画像内カット禁止の鉄則を破ることが必要になる。システムは要素構造を無視して8000pxでハードカットを実行し、公開失敗を防ぐ。

---

### 二段階容量事前確認メカニズム

フルレンダリング完了後に初めて超過を発見する（レンダリングリソース無駄）ことを避けるため、二段階検出を実施：

#### 第1段階：推定確認（レンダリング前、ミリ秒レベル）

Markdownソースファイルに基づく高速統計推定：

| 要素タイプ | 推定高さ |
|----------|--------|
| 本文各行（920px幅/32pxサイズ、約28文字/行） | 56px（行高含む） |
| 各段落間隔 | 56px |
| 各H1見出し（前後間隔含む） | 90 + 108 + 40 = 238px |
| 各H2見出し（前後間隔含む） | 73 + 96 + 32 = 201px |
| リモート `http` ネットワーク画像 | **即座にエラーで傍受**（現バージョンはネットワーク画像の安定性予測をサポートしない） |
| ローカル相対パス画像 | 「**幅でスケール、拡大なし**」規則に従い：元画像幅 > 920px の場合 `実際高さ×(920/実際幅)` で計算；元画像幅 ≤ 920px の場合、実際の物理高さをそのまま使用（拡大なし）。ミリ秒レベルのヘッダー読み取り — 正確。 |
| 上下余白 | 108 + 88 = 196px |

**推定式**：

```
H_estimated = 196（固定余白）
            + Σ 本文行高さ
            + Σ 段落間隔
            + Σ 見出し高さ
            + Σ 埋め込み画像推定高さ
```

**事前確認閾値**：`H_estimated > 32000 × 0.75`（24000px）で警告を発出；`> 32000px` で即エラー、レンダリングをスキップ。

> 0.75係数は推定が保守的なことを反映し、実際のレンダリングに余裕を残すため。24000pxを超えた場合、レンダリング後の実際の高さが最終的な判断基準となる。

#### 第2段階：精密確認（レンダリング後）

- Playwrightのスクリーンショット完了後、実際のページ高さ `page.height` を読み取る
- `page.height > 32000px` の場合 → エラーを出力し、すべての一時ファイルを削除
- エラーメッセージには：実際の高さ、超過量、削減すべき文字数の推計を含める

---

### エラーメッセージ仕様

容量上限を超えた場合、システムは構造化エラーを出力：

```
ERROR [CAPACITY_EXCEEDED]
  実際のレンダリング高さ：36480px
  容量上限：              32000px（4枚 × 8000px）
  超過量：                4480px（約14%）
  削減推奨文字数：         約800〜1000文字（中国語本文推計）
  現バージョン：          自動トランケーションはサポートしない — ソースMarkdownを手動で削減してから再試行
  将来の機能：            将来のバージョンではAgent駆動のテキスト圧縮で容量制約に自動適合する予定
```

---

### 分割後の出力ファイル命名規則

```
{filename}_1of{N}.png
{filename}_2of{N}.png
...
{filename}_Nof{N}.png
```

例：入力ファイル `article.md`、3枚に分割 → `article_1of3.png`、`article_2of3.png`、`article_3of3.png`

---

### 視覚的連続性の処理

| 要素 | 処理方法 |
|-----|---------|
| カット位置の下余白 | 通常の下余白（88px）を保持、余分なインジケーターテキストを追加しない |
| カット位置の上余白 | 通常の上余白（108px）を保持、余分なインジケーターテキストを追加しない |
| ページ番号 | オプション：各画像の右下に「1/3」「2/3」などの小さなラベルを追加（24px、透明度40%） |
| ブランド識別 | 必要な場合、最初の画像のみに配置、後続の画像には繰り返さない |

> 現バージョンではページ番号は必須ではない — 実装者の裁量に委ねる。
