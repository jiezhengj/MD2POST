# MD2POST：Markdown 转长图渲染引擎技术架构方案

> 状态：设计中
> 最后更新：2026-03-29
> 关联文档：[`长图排版最佳实践.md`](./长图排版最佳实践.md)

---

## 一、系统定位与核心要求

MD2POST 定位为一款**高度确定性、排版专业级**的 Markdown 转长图工具，专为 X/Twitter、微博等社交媒体传播设计。

**核心技术要求：**
1. **中日文排字绝对正确**：必须原生支持 CJK 禁则处理（标点挤压、行首行尾折行限制）。
2. **像素级参数控制**：严格遵循《长图排版最佳实践》，实现字距、行高、段间距的精准渲染。
3. **AI Native 编排**：引入 Agent 负责无破坏性的前置决策（语言嗅探、主题推理），但不干预确定性渲染链路。
4. **多图智能切片**：针对社交媒体平台容量限制，实现基于语义边界的安全切图。

---

## 二、技术栈选型选型

由于 CJK 复杂排版机制的特殊性，放弃纯 Canvas/Pango 的手动计算方案，采用**无头浏览器截图**的黄金链路。

| 模块层级 | 技术栈 | 选型理由 |
|---|---|---|
| **核心语言** | **TypeScript (Node.js)** | 前端及浏览器自动化生态最繁荣，类型系统保障参数契约。 |
| **AST 解析** | **`markdown-it`** | 插件生态丰富，易于扩展对脚注、Ruby 拼音、代码高亮的支持。 |
| **渲染引擎** | **Playwright** | 驱动 Headless Chromium 渲染，内核自带最完美的中日文断字算法；比 Puppeteer 更快、架构更现代。 |
| **大模型框架** | **Vercel AI SDK + Zod** | 用于 Agent 阶段，强制输出严格的 Zod Schema 校验的 JSON TypoGraphyConfig。 |
| **图像后处理** | **`sharp`** | Node.js 下性能最强的 C++ 图像处理库，负责最终的切图、格式优化和输出。 |

---

## 三、渲染流水线架构（Pipeline）

系统核心工作流分为“预处理（脑）”与“渲染执行（肌肉）”两个阶段。遵循“动静平衡观/确切性剥离”。

### 阶段一：Agent 智能决策引擎

**执行者**：大语言模型（如 Claude / GPT-4o）
**职责**：文本属性的极速判别，输出排版入参契约。

1. **输入提取**：截取 Markdown 源文件前 800 字符。
2. **多维嗅探**：
   - 嗅探主要语言（决定基础字号基准：中 32px / 日 30px / 英 28px）。
   - 嗅探内容类型特征（科技 / 人文 / 情感）。
3. **参数生成**：
   - 组装为符合 `TypographyConfig` 接口形态的 JSON。
   - 映射到预设的三大主题令牌（冷白 / 暖白 / 奶油黄）。

### 阶段二：DOM 组装与预检（双阶段熔断）

**执行者**：Node.js 逻辑层
**职责**：拦截异常请求，避免无效的端到端浏览器渲染开销。

1. **高度粗估计算**：
   - 解析 Markdown AST。
   - 按照《最佳实践》高度公式，叠加行高、自然段进行物理时长估算。
   - 针对图片，引入 `image-size` 实时探知**本地相对路径图片**的面宽数据。遵循**“只缩不放”**原则计算：宽度超 920px 时等比缩小高度，不足则按物理实际高度排列，杜绝失真漏洞。
   - **拦截协议**：如检测到带有 `http` / `https` 的网络图片，直接在此前置验证阶段抛出致命错误阻断流水线。
   - **第一级熔断**：总估算值若超过 24000px，抛出容量阈值警告（Warn）。若估算超过 32000px，直接熔断（`CAPACITY_EXCEEDED` 错误）。
2. **HTML/CSS 拼装**：
   - 将 Markdown 转换为 HTML DOM。
   - 注入包含 Design Tokens 的 CSS 模板，将 Agent 输出的主题色系化作 `:root` 变量（CSS Variables）。

### 阶段三：光栅化与物理切割

**执行者**：Playwright + Sharp
**职责**：全景截图与基于语义边界的物理裁切。

1. **浏览器渲染与截图**：
   - Playwright 打开虚拟页面，加载本地 `Noto Fonts`。
   - 等待所有 `<img>` 资源加载完成。
   - 执行 `page.screenshot({ fullPage: true })` 获取全景 PNG 内存 Buffer。
2. **第二级熔断**：
   - 读取实际截图高度。如果 Height > 32000px，触发第二级确切熔断，销毁任务。
3. **语义切片处理与越界降级**：
   - 如果 Height > 8000px 进行张数计算：`N = ceil(Height / 8000)`。
   - 在 Playwright 环境中使用 JS 执行**DOM 探测**（基于元素 `getBoundingClientRect()`），寻找 H1/H2、段落间隙的 `y` 坐标。
   - 获取最佳切点 `[y1, y2, ...]` 返回给 Node。
   - **越界死锁解除（Oversized Fallback）**：若某单体元素过巨掩盖了探测窗口，系统将优先回退到图片上侧边界开刀（此时单图高度变短）。除非目标突破平台 10k 红线，才触发降阶粗暴切除。
4. **图像输出**：
   - `sharp(buffer).extract({ top, left, width, height })` 按照切点数组输出多张 `_1of3.png` 终图。

## 四、核心数据契约（Schema 示例）

Agent 层生成的 `TypographyConfig` 必须受到 Zod Schema 严格约束。这是动静两个世界的隔离墙：

```typescript
// TypographyConfig Schema 契约
interface TypographyConfig {
  meta: {
    language: "zh-CN" | "ja" | "en";
    detectedType: "tech" | "humanities" | "emotion";
  };
  designTokens: {
    // 物理缩放系数值
    canvasWidth: 1080;
    baselineFontSize: number; // 32, 30 或 28
    
    // 主题色配置策略
    palette: {
      backgroundColor: string;
      primaryText: string;
      secondaryText: string;
      borderColor: string;
    };
  };
}
```

## 五、关键技术难点对策

### 1. 跨平台系统级字体一致性保障
- **问题**：Docker 或不同 OS 下默认中文字体错散。
- **方案**：应用层将 `Noto Sans CJK` 作为静态资源打入工程，浏览器内通过 `@font-face` 直接指向本地物理路径挂载，剥离对宿主机环境的字体依赖。

### 2. 长图切片的文字被拦腰切断防范
- **问题**：单纯基于高度等比切图一定会砍断某行文字或图片。
- **方案**：必须利用 Playwright 强大的环境优势。在 `screenshot` 之前，通过 `page.evaluate` 在页面内注入一个探针脚本。
  - 脚本遍历 DOM 树内所有块级元素（`h1...h5, p, img, pre`）。
  - 计算其物理占据的 `top` 与 `bottom` 取值范围。
  - 在理想切点 ±15% 区间内，反推查找到最近的元素间隙。并将该精确像素 Y 坐标发送给外层的 `sharp` 库执行硬件级分割。

### 3. I/O 阻塞防护与多核并行
- **问题**：大流量请求下，Playwright 环境起停沉重且耗时，极易假死。
- **方案**：使用 `generic-pool` 维持一个小规模的 Browser Context 连接池。并在 Python / Node 的通信缓冲区（tty）严防注入大文本对象代码段，改为全链路基于 `/tmp/xxx` 的物理文件路径传递通信。

---

*该方案确立了由“Agent 轻决策 + 头端硬检查 + 浏览器精准渲染 + 语义坐标裁切”构建的企业级无损流水线，彻底隔离因大模型错乱可能导致的排版崩塌风险。*
