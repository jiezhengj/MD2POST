# MD2POST：技術アーキテクチャ方案

> 状態：開発中
> 最終更新：2026-04-26
> 関連ドキュメント：[長編画像組版ベストプラクティス](./best-practices.ja.md)

---

## I. システム定位とコア要件

MD2POST は**高度な決定性・プロ組版グレード**の Markdown 長編画像ツールとして位置づけられており、X/Twitter・微博などのソーシャルメディア配信用に設計されています。

**コア技術要件：**
1. **CJK 組版の絶対的正確性**：禁則処理（句読点圧縮、行頭・行末折り返し制限）のネイティブサポートが必須。
2. **ピクセルレベルのパラメータ制御**：《長編画像組版ベストプラクティス》に厳密に従い、文字間隔・行高・段落間隔の精確なレンダリングを実現。
3. **AI Native オーケストレーション**：Agent が非破壊的な前置決定（言語嗅探・テーマ推論）を担当するが、決定論的レンダリングパイプラインには介入しない。
4. **マルチ画像スマートスライシング**：ソーシャルメディアプラットフォームの容量制限に対し、セマンティック境界ベースの安全な切り取りを実現。

---

## II. 技術スタック選定

CJK 組版の複雑さから、純粋な Canvas/Pango による手動計算方式を放棄し、**ヘッドレスブラウザスクリーンショット**のゴールデンパイプラインを採用。

| モジュール層 | 技術スタック | 選定理由 |
|------------|-----------|---------|
| **コア言語** | **TypeScript (Node.js)** | フロントエンド・ブラウザ自動化エコシステムが最も豊富。型システムがパラメータ契約を保証 |
| **AST パース** | **`markdown-it`** | プラグインエコシステムが豊富。注釈・Ruby・コードハイライトの拡張が容易 |
| **レンダリングエンジン** | **Playwright** | Headless Chromium を駆動。カーネルに最完全な CJK 禁則アルゴリズムを内蔵。Puppeteer より高速で現代的なアーキテクチャ |
| **LLM フレームワーク** | **Vercel AI SDK + Zod** | Agent ステージで使用。Zod スキーマで厳密に検証された JSON TypographyConfig の出力を強制 |
| **画像後処理** | **`sharp`** | Node.js で最高性能の C++ 画像処理ライブラリ。最終的な切り取り・フォーマット最適化・出力を担当 |

---

## III. レンダリングパイプラインアーキテクチャ

システムのコアワークフローは「前処理（脳）」と「レンダリング実行（筋肉）」の2フェーズに分かれており、「動静バランス観 / 決定性分離」の原則に従います。

### フェーズ1：Agent インテリジェント意思決定エンジン

**実行者**：大規模言語モデル（Claude / GPT-4o など）
**職責**：テキスト属性の超高速分類、組版入力パラメータ契約の出力。

1. **入力抽出**：Markdown ソースファイルの先頭800文字を取得。
2. **多次元嗅探**：
   - 主要言語を嗅探（基本フォントサイズを決定：中国語 32px / 日本語 30px / 英語 28px）。
   - コンテンツタイプの特性を嗅探（テクノロジー / 人文 / エモーション）。
3. **パラメータ生成**：
   - `TypographyConfig` インターフェース形態の JSON に組み立て。
   - 3種類のプリセットテーマトークン（クールホワイト / ウォームホワイト / クリームイエロー）にマッピング。

### フェーズ2：DOM アセンブリと事前検査（二段階ヒューズ）

**実行者**：Node.js ロジック層
**職責**：異常リクエストを傍受し、無駄なエンドツーエンドのブラウザレンダリングコストを回避。

1. **大まかな高さ推定**：
   - Markdown AST をパース。
   - ベストプラクティスの高さ計算式に従い、行高と段落を積み重ねて物理的な高さを推定。
   - 画像については `image-size` を使用してローカル相対パス画像の幅データをリアルタイムで取得。**「縮小のみ、拡大なし」**原則に従い計算。
   - **傍受プロトコル**：`http` / `https` のネットワーク画像が検出された場合、この前置検証段階で即座に致命的エラーを投げてパイプラインを中断。
   - **第一レベルヒューズ**：総推定値が 24000px を超えた場合、容量閾値警告を発出。32000px を超えた場合、即座にヒューズ（`CAPACITY_EXCEEDED` エラー）。
2. **HTML/CSS アセンブリとベクターマウント**：
   - Markdown を HTML DOM に変換。この段階で `markdown-it-mark`、`markdown-it-sup`、`markdown-it-sub`、`markdown-it-task-lists` の4つの公式プラグインをロードし、ハイライト・上下付き文字・Checklist 等の高度な組版構文をサポート。
   - **Mermaid 特化インターセプト**：エンジンは `markdown-it` ネイティブの `.renderer.rules.fence` 書き換えアルゴリズムを使用。`info === 'mermaid'` のコードブロックを嗅探した瞬間、ダーク背景のコードコンテナに押し込む代わりに、ローカルフォークの `beautiful-mermaid`（`vendor/beautiful-mermaid/` に配置）を呼び出してレンダリング待機なしの高精度 SVG ベクター画像文字列をコンパイルし、DOM にシームレスに挿入。このフォークには **ELK レイアウトエンジン設定を内置**し、フローチャートの垂直レイアウト（TD/TB）を強制することで、Issue #83 の TD/TB コマンドが無視される問題を完全に修正。`beautiful-mermaid` は初回使用前に `npm run build:mermaid` でコンパイルする必要がある。
   - Design Tokens を含む CSS テンプレートを注入し、Agent 出力のテーマを `:root` 変数（CSS Variables）として色系化。生成された Mermaid も `color-mix` ベースの変数（`bg`・`fg` など）によりテーマの色彩質感を自然に継承。
   - CSS 内部にすべての高度機能のビジュアルデザインプリセットを同期統合（例：丸角チェックボックス・精密整列の上付き文字・SVG の強制全幅伸展アルゴリズム）。

### フェーズ3：ラスタライズと物理切り取り

**実行者**：Playwright + Sharp
**職責**：全画面スクリーンショットとセマンティック境界ベースの物理切り取り。

1. **ブラウザレンダリングとスクリーンショット**：
   - Playwright が仮想ページを開き、ローカル `Noto Fonts` を読み込む。
   - すべての `<img>` リソースの読み込み完了を待機。
   - `page.screenshot({ fullPage: true })` を実行してフルページ PNG メモリバッファを取得。
2. **第二レベルヒューズ**：
   - 実際のスクリーンショット高さを読み取る。Height > 32000px の場合、第二レベル精密ヒューズを発動してタスクを破棄。
3. **セマンティックスライシングと越境降級**：
   - Height > 8000px の場合、タイル数を計算：`N = ceil(Height / 8000)`。
   - Playwright 環境で JS **DOM プロービング**（`getBoundingClientRect()` ベース）を使用し、H1/H2 要素と段落間隔の `y` 座標を取得。
   - 最適カットポイント `[y1, y2, ...]` を Node に返す。
   - **越境デッドロック解除（Oversized Fallback）**：単一の巨大要素が検出ウィンドウを遮蔽している場合、システムは優先的に画像の上側境界でカット（このときの単一タイル高さが短くなる）。ターゲットがプラットフォームの10k ハード上限を突破する場合のみ、強制粗カットを発動。
4. **画像出力**：
   - `sharp(buffer).extract({ top, left, width, height })` がカットポイント配列に基づいて複数の `_1of3.png` 最終画像を出力。

## IV. コアデータ契約（スキーマ例）

Agent 層が生成する `TypographyConfig` は Zod スキーマによる厳密な制約を受けなければならない。これが動的世界と静的世界の間の隔離壁：

```typescript
// TypographyConfig スキーマ契約
interface TypographyConfig {
  meta: {
    language: "zh-CN" | "ja" | "en";
    detectedType: "tech" | "humanities" | "emotion";
  };
  designTokens: {
    // 物理スケーリング係数値
    canvasWidth: 1080;
    baselineFontSize: number; // 32、30、または28

    // テーマカラー設定戦略
    palette: {
      backgroundColor: string;
      primaryText: string;
      secondaryText: string;
      borderColor: string;
    };
  };
}
```

## V. 主要技術課題と対策

### 1. クロスプラットフォームシステムフォント一貫性の保証
- **問題**：Docker や異なる OS 環境下でデフォルトの CJK フォントが欠損・散在する可能性がある。
- **対策**：`Noto Sans CJK` をプロジェクトの静的アセットとしてバンドルする。ブラウザ内では `@font-face` でローカル物理パスを直接指定してマウントし、ホストマシンのフォント環境への依存を排除。

### 2. 長編画像スライシング時のテキスト分断防止
- **問題**：単純な高さ比例切り取りは必ずテキスト行や画像を途中で切断してしまう。
- **対策**：Playwright の強力な環境優位性を活用。`screenshot` の前に `page.evaluate` を通じてページ内にプロブスクリプトを注入。
  - スクリプトがすべてのブロックレベル DOM 要素（`h1...h5, p, img, pre`）を走査。
  - 物理的な `top` と `bottom` 座標範囲を計算。
  - 理想カットポイントの ±15% ウィンドウ内で最も近い要素間隙を逆算し、その正確なピクセル Y 座標を外部の `sharp` ライブラリに送ってハードウェアレベルの分割を実行。

### 3. I/O ブロッキング保護とマルチコア並列処理
- **問題**：高トラフィックリクエスト下では Playwright 環境の起動/停止が重く時間がかかり、フリーズしやすい。
- **対策**：`generic-pool` を使用して小規模な Browser Context 接続プールを維持。Python/Node の通信バッファ（TTY）への大規模テキストオブジェクトコードセグメントの注入を厳禁し、`/tmp/xxx` の物理ファイルパスに基づく全チェーン通信に切り替える。

---

*このアーキテクチャは「Agent 軽量意思決定 + フロントエンドハードチェック + ブラウザ精密レンダリング + セマンティック座標切り取り」によるライトグレードの無損パイプラインを確立し、LLM の誤りによる組版崩壊リスクを徹底的に隔離します。*
